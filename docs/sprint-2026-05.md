## Workstream 1 — Extended thinking + parallel tool calls

**Estimate**: ~1 day total (0.5 + 0.5).
**Touches**: `src/lib/agents/orchestrator.ts`, `src/lib/agents/prompts.ts`, `src/lib/pricing.ts`, tests.
**Risk**: cost regression if budgets are mis-set; SDK shape changes between Opus versions.

### What changes

Today: every wake is a sequential loop of `messages.create` calls. The model sees its tool result, decides next move, calls one tool, repeats. Up to 16 turns. No `thinking` block.

After: the wake fires with two model-side capabilities enabled.

1. **Extended thinking** — `thinking: { type: 'enabled', budget_tokens: N }` on the request. Opus 4.7 gets a private scratchpad to reason for 5-30 s before each tool call. Default budget 8 000 tokens; configurable per-wake.
2. **Parallel tool use** — let Opus emit 2-5 `tool_use` blocks in a single response, all of which execute concurrently in our orchestrator before the next turn fires. Today we always handle one tool per turn; the loop already iterates `content` blocks but execution is serial.

### Implementation

**1.1 — Enable thinking (half-day)**

- Add `THINKING_BUDGET_TOKENS = 8_000` constant in `src/lib/agents/orchestrator.ts`.
- Pass `thinking: { type: 'enabled', budget_tokens: THINKING_BUDGET_TOKENS }` on every `messages.create` for the agent path. **Not** for meeting / wizard / Burrybot calls — those don't need it.
- Update `estimateCostUsd` in `src/lib/pricing.ts` to account for `thinking_tokens` separately. Anthropic bills these at output-token rates; capture from `usage.output_tokens` (thinking is folded into output by the SDK).
- Surface in `AgentRun.costUsd` accounting so the cost-summary on `/analytics` reflects reality.
- Mode override: `AGBRO_THINKING_DISABLED=true` env var bypasses thinking — gives ops an emergency knob.

**1.2 — Enable parallel tool calls (half-day)**

- Confirm SDK default in current `@anthropic-ai/sdk` version. As of v4.x, `disable_parallel_tool_use` is a `tool_choice` flag; default is parallel-on.
- Audit the orchestrator's tool-execution loop. Today it iterates `content` blocks but executes them sequentially. Switch to `Promise.all(toolUses.map(runTool))` so concurrent tool calls actually execute concurrently.
- Sequence guard: trade-mutating tools (`place_trade`, `place_option_trade`, `add_to_watchlist`, `acknowledge_thesis_review`) must NOT execute in parallel with each other or with reads of the same symbol. Add a small dependency-graph check: if any block in the batch is mutating, run that batch serially. Reads-only batches go parallel.
- Update agent prompt section 1: "When researching multiple watchlist names, call `research_perplexity` / `refresh_fundamentals` for ALL of them in a single response — do not chain one at a time. The orchestrator runs them in parallel."

### Tests

- Extend `pricing.test.ts` with thinking-token cost cases (Opus thinking rate matches output rate).
- Add `orchestrator.test.ts` that mocks Anthropic + verifies parallel batch execution: 3 read-only `tool_use` blocks → all 3 helpers called concurrently (assert via `Promise.all` race using `vi.fn` invocation order).
- Pin the dependency-graph guard: a batch with one mutating + two reads serializes.

### Acceptance

- Single wake on a watchlist with 5 names completes in **≤ 60 % of the prior median wake time** because research now batches.
- Cost per wake stays within ±15 % of prior median (thinking adds tokens but parallel reduces total turns).
- Property: every successful run has at least one `thinking` block in `AgentDecision.rawResponse` (when run with `AGBRO_THINKING_DISABLED!=true`).

## Workstream 2 — Post-mortem learning loop

**Estimate**: 2-3 days.
**Touches**: `prisma/schema.prisma` (one new field), `src/lib/agents/post-mortem.ts` (new), `src/lib/agents/tools.ts`, `src/lib/agents/schemas.ts`, prompt update, tests.
**Risk**: duplicate post-mortem entries clogging the brain; agent drifts into self-flagellation instead of decision-making.

### What changes

Today: every closed `Trade` has a `realizedPnlCents` and a `thesis` string. Every buy that placed had a `BrainEntry` with the rationale (sometimes — depends on the path). Nobody connects the two. The agent never reads its own track record.

After: a weekly post-mortem run that:
1. Pulls trades closed in the last 7 days
2. For each, finds the original buy thesis (via `BrainEntry.relatedSymbols` + `BrainEntry.sourceRunId`)
3. Asks Opus: was this thesis correct? If not, why not — and is the lesson generalizable?
4. Writes a new `BrainEntry` with `kind='post_mortem'`, `category='memory'`, `confidence='high'` if the loss / gain was decisive, `medium` otherwise
5. If the thesis was wrong AND the agent identifies a flawed reasoning step, sets `supersededById` on the original thesis entry to point at the post-mortem
6. Future wakes read the post-mortems via `read_brain` (already supported) — agent sees its own history of mistakes

### Implementation

**2.1 — Schema (half-day)**

The `BrainEntry` model already has `supersededById` self-reference. One small addition:

```prisma
model BrainEntry {
  // ... existing fields
  postMortemTradeIds  String[]  @default([])
  // ^ trades this post-mortem entry analyzed; lets the UI surface
  // "lessons from your KO sale" when viewing a closed Trade
}
```

**2.2 — Helper module (1 day)**

`src/lib/agents/post-mortem.ts` exports:

```ts
export type PostMortemContext = {
  trade: Trade;
  originalThesis: BrainEntry | null;  // the buy-time hypothesis
  relatedNotes: BrainEntry[];          // research notes on the symbol
  meetingDecisions: PolicyChange[];    // anything the firm decided that touched this symbol
};

export async function gatherPostMortemContext(userId: string, tradeId: string): Promise<PostMortemContext>;
export async function runPostMortem(userId: string, lookbackDays: number): Promise<PostMortemResult[]>;
```

`runPostMortem` calls Opus once per closed trade with the context, parses the response, writes the new `BrainEntry`, optionally sets `supersededById` on the prior thesis. Cap of 5 trades per run to keep cost bounded.

**2.3 — Tool exposure (half-day)**

New tool `run_post_mortem(lookbackDays?: number)`. Capped to 1 call per agent run. System prompt update: "On the first wake of any Monday, call `run_post_mortem(7)` before any new research. The lessons it writes feed `read_brain` immediately."

**2.4 — Tests (half-day)**

- Pure-function tests on the context-gathering query (mocked Prisma).
- Property test: post-mortem never creates an entry that supersedes itself or supersedes an entry from a different symbol.
- Integration-style test: feed a synthetic Trade + thesis pair, verify the resulting post-mortem entry shape.

### Acceptance

- Monday wakes (or `run_post_mortem` triggered from CLI for testing) produces post-mortem entries with non-empty bodies on every closed trade in the window.
- `read_brain({ kind: 'post_mortem' })` returns rows.
- Visiting `/brain` shows post-mortems with a distinct badge (one-line UI add).
- Property: zero duplicate post-mortems for the same `(tradeId, sourceRunId)` pair.

## Workstream 3 — Walk-forward backtest validation

**Estimate**: ~3 days.
**Touches**: `src/lib/backtest/walk-forward.ts` (new), `src/app/backtest/walk-forward/page.tsx` (new), `prisma/schema.prisma` (one new model), tests.
**Risk**: performance regression on long windows; the worst case is a strategy that LOOKS GREAT in single-run but craters out-of-sample, which is exactly what we need to expose.

### What changes

Today: `/backtest` runs a single strategy from `startDate` to `endDate`. Single equity curve, single set of metrics. We have **no way to know** whether the strategy's results are real or curve-fit to the picked window.

After: walk-forward harness that splits a long history into rolling train/test windows and reports per-window metrics + an aggregate. A strategy that performs reliably across rolling windows is real; one that only works on a curated window stands exposed.

### Implementation

**3.1 — Walk-forward runner (1.5 days)**

`src/lib/backtest/walk-forward.ts`:

```ts
export type WalkForwardConfig = {
  strategyKey: StrategyKey;
  totalStart: Date;            // e.g. 2010-01-01
  totalEnd: Date;              // e.g. 2024-12-31
  windowMonths: number;        // length of each test window, e.g. 24
  stepMonths: number;          // how far to slide between windows, e.g. 12
  universe: string[];
};

export type WalkForwardWindow = {
  start: Date;
  end: Date;
  metrics: BacktestMetrics;     // CAGR, maxDrawdown, sharpe, hitRate
  spyComparison: { cagr: number; alpha: number };
};

export async function runWalkForward(cfg: WalkForwardConfig): Promise<{
  windows: WalkForwardWindow[];
  aggregate: { medianCagr: number; medianMaxDrawdown: number; consistency: number };
}>;
```

**Note** on "training": our strategies are deterministic rule-based; there's no parameters to fit on a training slice. So walk-forward in our case is **pure out-of-sample validation across rolling windows**. The "training" period is just whatever lookback the strategy's rules need (e.g. 30-day drawdown calc).

**3.2 — Persistence (half-day)**

```prisma
model WalkForwardRun {
  id              String   @id @default(cuid())
  userId          String
  user            User     @relation(...)
  strategyKey     String
  config          Json     // serialised WalkForwardConfig
  windows         Json     // array of WalkForwardWindow
  aggregate       Json
  startedAt       DateTime @default(now())
  completedAt     DateTime?
  status          String   // 'running' | 'completed' | 'errored'
  errorMessage    String?  @db.Text
}
```

Lets users compare prior walk-forwards across config changes.

**3.3 — UI (1 day)**

`/backtest/walk-forward` page:
- Form: pick strategy, total date range, window months, step months, universe.
- Results: a heatmap (rows = windows, columns = metrics) + a stacked bar chart of per-window CAGR vs SPY's CAGR.
- "Consistency" metric: `min(metric) / max(metric)` across windows for each strategy. Closer to 1 = robust; closer to 0 = curve-fit.
- "Run again" button to re-run with different config.

### Tests

- Pure-function test on the window-splitting logic: `splitWindows(2015-01-01, 2020-12-31, 24, 12)` returns `[[2015-01..2017-01], [2016-01..2018-01], ...]`.
- Property test: `windows[i].end > windows[i].start` and `windows[i+1].start ≥ windows[i].start`.
- Mutation-verified: deliberately break the step calculation (off-by-one), confirm tests catch it.
- Backtest engine reused as-is; the harness is purely compositional, no new strategy logic.

### Acceptance

- Run walk-forward on Buffett Core 2015-2024, window=24mo, step=12mo. Result: 7 windows, each with CAGR + drawdown.
- Consistency metric is reported for each preset strategy on the `/backtest/walk-forward` page.
- A regression that broke a strategy's exit rule shows up as a sharp consistency drop in one window — proves the harness works.

## Sequencing

Run them in this order:

1. **Week 1, day 1** — Workstream 1 (extended thinking). Half day.
2. **Week 1, day 1-2** — Workstream 1 (parallel tools). Half day.
3. **Week 1, day 2-4** — Workstream 2 (post-mortem). 2-3 days.
4. **Week 1, day 5 → Week 2, day 1** — Workstream 3 (walk-forward). 3 days.

After workstream 1: the cost of every subsequent agent run *is* a workstream-1 metric. Worth letting it sit a day before moving on so we have data on how parallel + thinking actually affects spend.

After workstream 2: the agent's writing post-mortems but they only help if it READS them. Workstream 1's parallel tool calls let it pull both `read_brain({ kind: 'post_mortem' })` and `read_brain({ kind: 'principle' })` simultaneously, which is the unlock.

After workstream 3: we have evidence — for the first time — that the strategies actually hold up out-of-sample. If they don't, we change the strategies before any real-money path.
